# Дорожная карта: привязка Telegram к пользователям Kanboard

## Цель
Сделать удобное и безопасное сопоставление пользователей Kanboard с Telegram-адресатами без правок ядра Kanboard и без ломки уже работающих уведомлений по задачам и подзадачам.

## Принципы
- Существующий рабочий контур уведомлений сохраняем.
- Изменения делаем эволюционно, небольшими шагами.
- Любые изменения на Raspberry Pi вне кода репозитория заранее обсуждаем и делаем с бэкапом.

## План работ
1. API привязок в `worker/bot_service`:
   - `GET /bindings`
   - `POST /bindings/upsert`
   - `POST /bindings/unbind`
   - `POST /bindings/test`
2. Страница управления в плагине `AssigneeNotify`:
   - список пользователей Kanboard
   - статус привязки
   - действия: привязать, отвязать, тест.
3. Пользовательский сценарий self-service:
   - одноразовый код в Kanboard
   - команда боту `/bind <code>`.
4. Проверка и стабилизация:
   - ручные проверки из web UI
   - проверка логов plugin/bot
   - фиксация кейсов и регрессий.

## Риски и контроль
- Риск: несовпадение id пользователя и привязки.
  - Контроль: явная проверка соответствия в UI и тест-сообщение.
- Риск: изменения в окружении сервера не в репо.
  - Контроль: перед правками делаем бэкап файлов и записываем шаги отката.
- Риск: случайная потеря runtime-данных при деплое.
  - Контроль: не деплоить `.env`, `*.sqlite`, `*.log` (уже учтено в скрипте).

## Критерии готовности (MVP)
- Админ может управлять привязками без SQL.
- Пользователь может привязать Telegram без ручного ввода `chat_id`.
- Тест-уведомление отправляется из UI.
- Уведомления по задаче и подзадаче продолжают работать.

- 2026-03-14: Исправлен баг формы теста/сохранения на странице Telegram bindings.
  - Ошибка: при нажатии `Test` возвращалось `Invalid user id`.
  - Причина: в контроллере использовался `Request::getIntegerParam()`, который в Kanboard читает только `$_GET`, тогда как форма отправляет `kanboard_user_id` через `POST`.
  - Решение: переход на чтение данных из `Request::getValues()` (POST + CSRF), добавлена явная проверка положительного `kanboard_user_id`.
- 2026-03-14: Добавлен контроль “теневых” привязок.
  - Что сделано: на странице `Telegram bindings` добавлен блок `Stale bindings` (неактивные/удаленные пользователи) и массовая кнопка `Deactivate all stale bindings`.
  - Цель: чтобы привязки ушедших сотрудников не оставались скрыто активными.
- 2026-03-14: Реализован self-service для обычных пользователей.
  - Что сделано в Kanboard: страница `User profile -> Telegram binding`, генерация одноразового кода, тест-сообщение.
  - Что сделано в bot-service: endpoint `POST /api/v1/bindings/token/create`, таблица `binding_tokens`, poller Telegram `getUpdates` и обработка команды `/bind <code>`.
  - Ограничение по процессу: при активной привязке код не перевязывает на новый chat_id (ожидается отвязка админом).
- 2026-03-14: Исправлен 500 на странице self-service (`Telegram binding`) для обычных пользователей.
  - Ошибка: `Call to undefined method Kanboard\Core\User\UserSession::isCurrentUser()`.
  - Причина: в контроллере использован метод, которого нет в `UserSession` текущей версии Kanboard.
  - Решение: проверка текущего пользователя заменена на сравнение `userSession->getId()` с `user['id']`.
- 2026-03-14: Исправлен UX-дефект генерации bind-кода на странице self-service.
  - Симптом: при нажатии `Generate bind code` запрос выполнялся, но пользователь визуально не видел результат.
  - Причина: код выводился только через flash-сообщение, которое на пользовательском layout легко пропустить.
  - Решение: после успешной генерации код и `expires_at` рендерятся в явном блоке на самой странице (`Bind code generated`, команда `/bind <code>`).
- 2026-03-14: Добавлена защита от коллизий `telegram_chat_id` между разными пользователями.
  - Симптом: один Telegram-чат можно было привязать к нескольким пользователям Kanboard.
  - Риск: уведомления разных пользователей приходили в один и тот же чат.
  - Решение: при `upsert` и self-bind (`/bind`) вводится проверка уникальности активного `telegram_chat_id`; при конфликте возвращается `status=chat_already_bound` и привязка не меняется.
