Нужно реализовать первый рабочий этап собственной системы уведомлений для Kanboard и Telegram.

Общая идея:
Система состоит из двух независимых компонентов:
1. Kanboard plugin
2. отдельный bot-service

На данном этапе нужен только базовый end-to-end pipeline:
изменился ответственный по задаче в Kanboard -> плагин фиксирует событие -> плагин отправляет команду bot-service -> bot-service отправляет обезличенное сообщение нужному пользователю в Telegram.

Никакой сложной бизнес-логики сейчас не делать.
Не реализовывать:
- повторные напоминания
- рабочее время
- эскалации
- подтверждение прочтения
- пользовательские настройки
- шаблонизацию
- несколько транспортов
- очереди сообщений
- retry/backoff orchestration
- сложный onboarding
- web UI

Нужен именно минимальный, но правильный каркас, который потом можно расширять без переписывания архитектуры.

==================================================
1. ЦЕЛЬ ПЕРВОГО ЭТАПА
==================================================

Нужно получить рабочую систему, в которой:

1. Kanboard plugin отслеживает реальное изменение ответственного у задачи.
2. Если назначен новый ответственный, plugin создает внутреннюю запись о событии.
3. Plugin отправляет HTTP POST в bot-service.
4. Bot-service принимает запрос, валидирует его, находит telegram_chat_id по kanboard_user_id и отправляет обезличенное сообщение.
5. Оба компонента логируют результат.
6. Если привязки Telegram нет, это не должно ломать систему: событие должно быть зафиксировано как failed / unmapped, но без падения.

==================================================
2. ЖЕСТКИЕ ГРАНИЦЫ ЭТАПА
==================================================

Что должно быть сделано:
- Kanboard plugin как отдельный plugin без модификации core
- отдельный bot-service
- минимальный HTTP контракт между ними
- минимальное хранение событий и результатов
- конфигурация через env / config
- README для локального запуска
- понятная структура каталогов
- минимальный onboarding через статическую mapping table допустим как временное решение

Что не должно быть сделано сейчас:
- никакой сложной доменной логики напоминаний
- никакого polling Kanboard API
- никакой двусторонней синхронизации
- никакого раскрытия деталей задачи в Telegram
- никакой интеграции с готовыми telegram plugins для Kanboard
- никакой правки ядра Kanboard
- никакой message queue типа RabbitMQ / Kafka / Redis streams
- никакой "enterprise architecture"

Если возникает неоднозначность, выбирать самый простой локально запускаемый вариант.

==================================================
3. ОБЩАЯ АРХИТЕКТУРА
==================================================

Разделить ответственность строго.

Kanboard plugin отвечает только за:
- обнаружение бизнес-события внутри Kanboard
- сравнение старого и нового ответственного
- фиксацию события в своей таблице
- отправку минимального HTTP запроса наружу
- логирование результата HTTP вызова

Bot-service отвечает только за:
- прием входящего HTTP запроса
- проверку авторизации и валидности payload
- поиск соответствия kanboard_user_id -> telegram_chat_id
- отправку сообщения через Telegram Bot API
- фиксацию результата отправки

Нельзя смешивать эти роли.
Bot-service не должен знать бизнес-логику Kanboard сверх минимального payload.
Plugin не должен знать деталей Telegram API.

==================================================
4. СОБЫТИЕ, КОТОРОЕ НАС ИНТЕРЕСУЕТ
==================================================

На этом этапе интересует только одно событие:
реальное изменение assignee у задачи.

Реальным изменением считать только случай, когда:
old_assignee_user_id != new_assignee_user_id

Случаи:
1. old = null, new = user_id -> это назначение, отправлять
2. old = user_a, new = user_b -> это переназначение, отправлять
3. old = user_a, new = user_a -> это не изменение, не отправлять
4. old = user_a, new = null -> это снятие ответственного, не отправлять, но можно логировать
5. old = null, new = null -> игнорировать

Эта логика должна быть реализована явно и изолированно, а не размазана по коду.

==================================================
5. ТРЕБОВАНИЯ К KANBOARD PLUGIN
==================================================

Нужно создать полноценный plugin для Kanboard со стандартной структурой.

Plugin должен:
- регистрироваться штатно через Plugin.php
- подключаться через стандартный механизм Kanboard plugins
- использовать hooks / events / listener подход, без изменения core
- иметь отдельный конфиг для bot-service URL и shared secret
- иметь собственную таблицу для журнала событий уведомлений
- иметь сервис/класс, отвечающий за определение факта изменения assignee
- иметь сервис/класс, отвечающий за отправку HTTP запроса в bot-service
- иметь сервис/класс, отвечающий за запись события в БД

Plugin должен работать так:
1. Получить сигнал об изменении задачи.
2. Извлечь старое и новое значение assignee.
3. Пропустить через comparator.
4. Если это назначение/переназначение, создать запись в таблице plugin notifications_events.
5. Сформировать минимальный payload.
6. Отправить POST в bot-service.
7. Обновить статус записи по результату.

Если bot-service недоступен, plugin не должен падать и не должен ломать сохранение задачи в Kanboard.
Ошибка должна быть залогирована и отражена в таблице событий.

7. PAYLOAD МЕЖДУ PLUGIN И BOT-SERVICE
==================================================

Передавать только минимально нужные данные.

Формат JSON:
{
  "event_id": "uuid",
  "event_type": "assignee_changed",
  "task_id": 123,
  "kanboard_user_id": 45,
  "occurred_at": "ISO8601 timestamp"
}

Можно дополнительно передавать:
- old_assignee_user_id
- new_assignee_user_id

Но не передавать ничего лишнего.

Строго запрещено передавать:
- title
- description
- project name
- creator name
- due date
- tags
- comments
- любые внутренние поля задачи, не нужные для доставки

==================================================
8. АВТОРИЗАЦИЯ МЕЖДУ PLUGIN И BOT-SERVICE
==================================================

Нужен простой shared secret.

Вариант:
HTTP header:
X-Webhook-Token: <secret>

Bot-service должен отвергать запросы без токена или с неправильным токеном.
Plugin должен брать токен из config/env.

Токен не логировать открытым текстом.

==================================================
9. ТРЕБОВАНИЯ К BOT-SERVICE
==================================================

Нужен отдельный небольшой сервис.
Без избыточной сложности.
Локально запускаемый на Linux / Raspberry Pi.

Язык и стек выбрать прагматично.
Главный критерий: простота чтения, простота запуска, простота ручной правки.
Не городить тяжелую архитектуру ради красоты.

Bot-service должен иметь:
1. HTTP endpoint приема событий от Kanboard plugin
2. проверку токена
3. валидацию JSON payload
4. lookup соответствия kanboard_user_id -> telegram_chat_id
5. отправку сообщения через Telegram Bot API
6. таблицу/хранилище mappings
7. таблицу/хранилище delivery logs

Если mapping не найден:
- вернуть корректный управляемый результат
- не падать
- зафиксировать в логе, что пользователь не привязан

==================================================
11. ПОВЕДЕНИЕ BOT-SERVICE ПРИ ПРИЕМЕ СОБЫТИЯ
==================================================

Алгоритм:
1. Проверить header token.
2. Проверить JSON body.
3. Проверить наличие обязательных полей.
4. Найти mapping по kanboard_user_id.
5. Если mapping отсутствует:
   - записать delivery_log со статусом unmapped
   - вернуть понятный ответ, например 404 или 200 с machine-readable status, выбрать один подход и описать в README
6. Если mapping найден:
   - отправить нейтральное сообщение в Telegram
   - записать результат
   - вернуть plugin понятный ответ

Сообщение пока одно и жестко заданное.
Например:
"Вам назначена задача. Зайдите в Kanboard."

Без деталей.
Без названия задачи.
Без id задачи в тексте.
Без имени проекта.
Без имени постановщика.

==================================================
12. ФОРМАТ ОТВЕТА BOT-SERVICE
==================================================

Нужен machine-readable JSON response.

Успех:
{
  "ok": true,
  "status": "delivered"
}

Если пользователь не привязан:
{
  "ok": false,
  "status": "unmapped"
}

Если ошибка в токене:
{
  "ok": false,
  "status": "unauthorized"
}

Если плохой payload:
{
  "ok": false,
  "status": "bad_request"
}

Если Telegram API вернул ошибку:
{
  "ok": false,
  "status": "telegram_error"
}

Plugin должен уметь это сохранять в свой журнал.

==================================================
14. ЛОГИРОВАНИЕ
==================================================

Логирование должно быть простым и читаемым.

Plugin логирует:
- факт обнаружения события
- создана ли запись
- отправлен ли HTTP запрос
- результат запроса
- текст ошибки без лишнего шума

Bot-service логирует:
- принят ли запрос
- прошел ли token check
- прошел ли payload validation
- найден ли mapping
- результат вызова Telegram API

Не логировать:
- токены
- чувствительные поля
- лишние дампы

18. КРИТЕРИИ ПРИЕМКИ
==================================================

Работа считается выполненной, если можно воспроизвести сценарий:

Сценарий A:
1. Есть задача без ответственного.
2. В Kanboard назначаем пользователя X.
3. Plugin фиксирует событие.
4. Plugin вызывает bot-service.
5. Bot-service находит mapping для X.
6. В Telegram приходит обезличенное сообщение.
7. В обоих компонентах есть записи в логах/таблицах.

Сценарий B:
1. У задачи уже был ответственный X.
2. Меняем на Y.
3. Уходит сообщение только Y.
4. Событие корректно фиксируется как переназначение.

Сценарий C:
1. У задачи ответственный X.
2. Сохраняем задачу без изменения assignee.
3. Никакого уведомления не уходит.
4. Лишних записей не создается.

Сценарий D:
1. Назначаем пользователя, для которого нет user_binding.
2. Plugin не падает.
3. Bot-service возвращает status=unmapped.
4. Это фиксируется в логах и таблицах.

Сценарий E:
1. Bot-service недоступен.
2. Kanboard продолжает работать.
3. Plugin фиксирует failed delivery, без фатальной ошибки для Kanboard.
